DOCKER Fundamentals

Docker packages an application and all its dependencies into a lightweight, portable container. Unlike VMs, containers share the host OS kernel — startup is milliseconds, not minutes.

Core concepts at a glance

Image

Read-only template built from a Dockerfile. Each instruction creates a layer. Images are immutable — you never modify them at runtime.

Container

A running instance of an image. Has a thin writable layer on top. Ephemeral by default — destroyed when stopped unless you use volumes.

Registry

Storage for images. Docker Hub is the default public registry. ECR, GCR, ACR, and Harbor are common enterprise options.

Docker Engine

Client-server architecture: docker CLI (client) → REST API → dockerd daemon → containerd → runc (OCI runtime).

DOCKER Dockerfile mastery

# ── Multi-stage build — production best practice ──────────────────
FROM node:20-alpine AS builder
WORKDIR /app

# Copy dependency manifests FIRST — separate layer, cached if unchanged
COPY package*.json ./
RUN npm ci --only=production

# Copy source and build
COPY . .
RUN npm run build

# ── Final stage — minimal runtime image ───────────────────────────
FROM node:20-alpine AS runtime
WORKDIR /app

# Non-root user — security hardening
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# Copy only built artifacts from builder stage
COPY --from=builder --chown=appuser:appgroup /app/dist ./dist
COPY --from=builder --chown=appuser:appgroup /app/node_modules ./node_modules

# Declare which port the app uses (documentation, not enforcement)
EXPOSE 3000

# Prefer ENTRYPOINT + CMD combination
ENTRYPOINT ["node"]
CMD ["dist/server.js"]

# Health check — critical for orchestrators
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD wget -q --spider http://localhost:3000/health || exit 1

DOCKER Layers & image caching

Layer caching rules

  • Copy dependency files (package.json, requirements.txt) before source code — dependencies change far less often
  • Each RUN command is a separate layer — chain related commands with && and clean up in the same layer
  • Use .dockerignore to prevent node_modules, .git, build outputs from invalidating layers
  • Multi-stage builds keep the final image small — build tools never reach production

DOCKER Networking

bridge (default)

Containers on the same bridge can communicate by container name. Isolated from host. Use for multi-container apps on a single host.

host

Container shares host network stack directly. No network isolation. Best for performance-critical apps that don't need isolation.

none

Container has no network interface. Complete network isolation. Use for batch jobs or security-sensitive workloads.

overlay

Spans multiple Docker hosts (Swarm mode). Containers on different hosts communicate as if on the same network. Built on VXLAN.

# Create a custom bridge network
docker network create --driver bridge app-network

# Containers on the same custom network resolve each other by name
docker run -d --name db --network app-network postgres:15
docker run -d --name api --network app-network \
  -e DB_HOST=db \
  my-api:latest

# Inspect network
docker network inspect app-network

DOCKER Security hardening

Docker security checklist

  • Always run containers as non-root: USER appuser in Dockerfile
  • Use minimal base images: alpine, distroless, or scratch
  • Scan images with docker scout, Trivy, or Snyk in CI
  • Never store secrets in environment variables or image layers — use secrets management
  • Set read-only filesystem: docker run --read-only
  • Drop all capabilities, add only what's needed: --cap-drop ALL --cap-add NET_BIND_SERVICE
  • Use --security-opt no-new-privileges:true to prevent privilege escalation
  • Pin image versions with digest: nginx@sha256:abc123...

KUBERNETES Architecture

Kubernetes is a container orchestrator. It manages scheduling, scaling, self-healing, networking, and storage across a cluster of nodes.

KUBERNETES Workloads

Deployment
Stateless apps. Manages ReplicaSets. Supports rolling updates and rollback. The most common workload type.
StatefulSet
Stateful apps (databases, Kafka). Stable pod names (pod-0, pod-1), ordered startup/shutdown, stable network IDs.
DaemonSet
Runs exactly one pod per node. Used for log collectors, monitoring agents, security scanners (Fluentd, Datadog, Falco).
Job / CronJob
Run-to-completion workloads. Jobs run once, CronJobs run on a schedule. Database migrations, batch processing, reports.
# Deployment with best practices
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
  labels: { app: api, version: v2 }
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate: { maxSurge: 1, maxUnavailable: 0 }
  selector:
    matchLabels: { app: api }
  template:
    metadata:
      labels: { app: api }
    spec:
      containers:
      - name: api
        image: myrepo/api:v2.1.0          # pin exact version, not :latest
        ports: [{ containerPort: 3000 }]
        resources:
          requests: { cpu: 100m, memory: 128Mi }
          limits:   { cpu: 500m, memory: 512Mi }
        readinessProbe:
          httpGet: { path: /health, port: 3000 }
          initialDelaySeconds: 10
          periodSeconds: 5
        livenessProbe:
          httpGet: { path: /health, port: 3000 }
          initialDelaySeconds: 30
          failureThreshold: 3
        envFrom:
        - configMapRef: { name: api-config }
        - secretRef:    { name: api-secrets }
      securityContext:
        runAsNonRoot: true
        runAsUser: 1000
        readOnlyRootFilesystem: true

KUBERNETES Services & networking

ClusterIP (default)

Internal-only virtual IP. Only reachable within the cluster. Used for service-to-service communication. Each service gets a DNS name: svc-name.namespace.svc.cluster.local.

NodePort

Exposes service on a static port (30000–32767) on every node's IP. Used for development or when you control the load balancer layer yourself.

LoadBalancer

Provisions a cloud load balancer (AWS ALB/NLB, GCP LB). The standard way to expose services publicly in managed Kubernetes clusters.

Ingress

HTTP/HTTPS routing with host/path rules, TLS termination, and virtual hosting — all through one IP address. Uses an Ingress Controller (nginx, Traefik, AWS ALB Ingress).

# Ingress with TLS termination and path routing
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  tls:
  - hosts: [api.example.com]
    secretName: api-tls
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /api     pathType: Prefix
        backend: { service: { name: api-svc, port: { number: 3000 } } }
      - path: /static  pathType: Prefix
        backend: { service: { name: static-svc, port: { number: 80 } } }

KUBERNETES Scheduling & pod placement

# Node affinity — prefer nodes with SSD, require us-east-1
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - { key: topology.kubernetes.io/zone, operator: In, values: [us-east-1a] }
  podAntiAffinity:                     # spread pods across nodes
    requiredDuringSchedulingIgnoredDuringExecution:
    - labelSelector:
        matchExpressions:
        - { key: app, operator: In, values: [api] }
      topologyKey: kubernetes.io/hostname

# Pod Disruption Budget — guarantee availability during maintenance
apiVersion: policy/v1
kind: PodDisruptionBudget
spec:
  minAvailable: 2                      # or maxUnavailable: 1
  selector: { matchLabels: { app: api } }

# Horizontal Pod Autoscaler — scale pods based on CPU/memory/custom metrics
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  scaleTargetRef: { apiVersion: apps/v1, kind: Deployment, name: api }
  minReplicas: 3
  maxReplicas: 20
  metrics:
  - type: Resource
    resource: { name: cpu, target: { type: Utilization, averageUtilization: 70 } }

    resource: { name: memory, target: { type: Utilization, averageUtilization: 80 } }
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # prevent thrashing

DOCKER Best practices summary

Dockerfile best practices

  • Use multi-stage builds — keep build tools out of production images
  • Use specific base image versions: node:20.11-alpine3.19 not node:latest
  • Copy package.json before source code to maximize layer cache reuse
  • Run as non-root user; set readOnlyRootFilesystem: true
  • Always include a HEALTHCHECK instruction
  • Use ENTRYPOINT for the executable, CMD for default arguments
  • Keep layers small — delete temp files in the same RUN that creates them
  • Scan every image for CVEs in CI before pushing to registry

KUBERNETES Best practices summary

K8s production checklist

  • Always set CPU/memory requests and limits — prevents noisy-neighbour starvation
  • Configure both readinessProbe and livenessProbe on every container
  • Never use :latest image tag in production manifests — pin exact versions
  • Use namespaces to isolate environments and enforce RBAC boundaries
  • Set a PodDisruptionBudget for every critical Deployment
  • Use podAntiAffinity to spread replicas across nodes/AZs
  • Encrypt Secrets at rest; use External Secrets Operator for secret injection
  • Enable NetworkPolicy to restrict inter-pod communication to what's needed
  • Use LimitRange and ResourceQuota per namespace to prevent resource abuse
  • Prefer Helm or Kustomize for managing YAML at scale

Interview Q&A — 3 years experience

Docker core concepts
Docker
Easy
What is the difference between a Docker image and a Docker container?
▼

An image is an immutable, read-only blueprint built from a Dockerfile. It contains the application code, runtime, dependencies, and configuration layered on top of a base OS. Images are stored in registries.

A container is a running instance of an image. Docker adds a thin writable layer on top of the read-only image layers for that specific container. Multiple containers can run from the same image simultaneously, each with their own writable layer.

Analogy: An image is like a class definition in OOP; a container is an object instantiated from that class.

docker build -t my-app:v1 .              # creates image
docker run -d --name c1 my-app:v1        # creates container from image
docker run -d --name c2 my-app:v1        # another container, same image
Docker
Easy
What is the difference between CMD and ENTRYPOINT in a Dockerfile?
▼

ENTRYPOINT defines the executable that always runs — it is not overridden by arguments to docker run. CMD provides default arguments to the entrypoint and can be overridden.

# ENTRYPOINT + CMD combination (recommended)
ENTRYPOINT ["node"]         # always runs node
CMD ["server.js"]           # default: node server.js

# Override CMD at runtime
docker run my-app worker.js  # runs: node worker.js

# CMD only (less preferred)
CMD ["node", "server.js"]   # entire command overridden by docker run args

Rule of thumb: Use ENTRYPOINT for the binary/runtime, CMD for the default script/argument. This gives callers flexibility to pass different scripts without rewriting the entrypoint.

Docker
Medium
Explain Docker layer caching and how you optimise a Dockerfile to take advantage of it.
▼

Docker caches each layer. When rebuilding, a layer is only re-executed if its instruction or any of its inputs have changed. If any layer is invalidated, all subsequent layers are rebuilt.

Optimisation strategy: Place instructions that change least frequently at the top. Copy dependency manifests before source code — dependencies change far less often than application code.

Slow (bad cache)

COPY . .              # copies everything
RUN npm install       # reinstalls on any code change

Fast (good cache)

COPY package*.json ./  # only manifests
RUN npm ci             # cached unless package.json changes
COPY . .               # source code last

Also use .dockerignore to prevent node_modules, .git, and build artefacts from being sent to the build context, which would also invalidate the source copy layer.

Docker
Medium
What is a multi-stage build and why should you use it?
▼

A multi-stage build uses multiple FROM statements in one Dockerfile. Earlier stages compile/build the application; the final stage copies only the necessary artefacts into a minimal runtime image.

Benefits:

  • Drastically smaller production images (a Go binary: 800MB with compiler → 10MB in scratch)
  • Build tools, compilers, and dev dependencies never reach production — reduces attack surface
  • One Dockerfile replaces separate build and runtime Dockerfiles
FROM golang:1.22 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 go build -o server ./cmd/server

FROM scratch            # zero-dependency final image
COPY --from=builder /app/server /server
COPY --from=builder /etc/ssl/certs /etc/ssl/certs
ENTRYPOINT ["/server"]

Docker
Hard
How does Docker networking work internally? Explain how containers communicate on a custom bridge network.
▼

Docker creates a Linux bridge interface (e.g. br-abc123) on the host for each custom network. Each container gets a veth (virtual Ethernet) pair: one end inside the container's network namespace, the other connected to the bridge.

Docker runs an embedded DNS resolver (127.0.0.11) inside each container. Containers resolve each other by name — Docker intercepts DNS queries and returns the container's IP on the same network.

Traffic between containers: container A → veth → bridge → veth → container B. Traffic to the internet: bridge → iptables NAT (MASQUERADE) → host NIC.

ip link show      # see bridge + veth pairs on host
ip route          # 172.17.0.0/16 via bridge for default network
iptables -t nat -L -n  # see MASQUERADE rules for outbound NAT
docker network inspect app-network  # see connected containers + IPs
Docker
Hard
How do you handle secrets in Docker in a secure way? What are the anti-patterns to avoid?
▼

Anti-patterns to avoid:

  • Hardcoding secrets in Dockerfile (ARG or ENV — visible in docker history)
  • Passing secrets via environment variables (visible in docker inspect, process list, logs)
  • Copying secret files into image layers (even if deleted in later layer, visible in earlier layer)

Correct approaches:

# 1. Docker secrets (Swarm) — mounted as tmpfs, never in image or env
echo "s3cr3t" | docker secret create db_pass -
docker service create --secret db_pass my-service
# mounted at /run/secrets/db_pass — readable only by container

# 2. BuildKit secret mount — available ONLY during build, not in final layer
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc npm install
DOCKER_BUILDKIT=1 docker build --secret id=npmrc,src=.npmrc .

# 3. Runtime: inject via orchestrator (K8s Secret, AWS Secrets Manager)
Kubernetes core concepts
Kubernetes
Easy
What is the difference between a Pod, Deployment, and ReplicaSet?
▼

A Pod is the smallest deployable unit in Kubernetes — one or more tightly-coupled containers sharing a network namespace and storage. Pods are ephemeral; if one dies, it is not automatically restarted.

A ReplicaSet ensures a specified number of identical Pod replicas are running at all times. It recreates pods that die. You rarely create ReplicaSets directly.

A Deployment manages ReplicaSets and adds rolling update and rollback capability. When you change the image or config in a Deployment, it creates a new ReplicaSet and gradually migrates traffic while keeping the old one around for rollback.

# Hierarchy
Deployment → ReplicaSet (current) + ReplicaSet (previous) → Pods

kubectl rollout history deployment/api
kubectl rollout undo deployment/api --to-revision=2
Kubernetes
Medium
What is the difference between readinessProbe and livenessProbe? When should you use each?
▼

A readinessProbe determines whether a pod is ready to receive traffic. Failing pods are removed from Service endpoints — they stay running but get no traffic. Used when an app needs warm-up time (loading config, connecting to DB, caching).

A livenessProbe determines whether a pod is still alive. A failing pod is restarted by kubelet. Used for detecting deadlocks, memory leaks, or processes that hang but don't crash.

Never use the same endpoint for both probes if they have different semantics. A heavy liveness check that fails due to downstream dependency will cause cascading restarts — use a shallow check for liveness, deeper dependency check for readiness.

readinessProbe:              # remove from LB if not ready
  httpGet: { path: /ready, port: 3000 }
  initialDelaySeconds: 10   # wait for app startup
  periodSeconds: 5
livenessProbe:               # restart if hung/deadlocked
  httpGet: { path: /health, port: 3000 }
  initialDelaySeconds: 30   # longer — don't restart during slow startup
  failureThreshold: 3
Kubernetes
Medium
Explain how Kubernetes Services work and how DNS resolution happens inside a cluster.
▼

A Service is an abstraction that gives a stable virtual IP (ClusterIP) and DNS name to a dynamic set of pods selected by labels. kube-proxy maintains iptables (or IPVS) rules on each node to load-balance traffic to healthy pod IPs.

Kubernetes runs CoreDNS (a cluster DNS server). Every pod's /etc/resolv.conf is configured to search the cluster DNS. A service named api in namespace production is reachable as:

api                                         # same namespace
api.production                              # short form
api.production.svc.cluster.local            # fully qualified
api.production.svc.cluster.local:3000       # with port

Headless Services (clusterIP: None) return individual pod IPs via DNS instead of a virtual IP — used for StatefulSets where each pod needs to be addressed directly (Cassandra, Kafka, etc.).

Kubernetes
Medium
What are resource requests and limits, and what happens if you don't set them?
▼

Requests are what the scheduler uses to decide which node can host a pod. A pod is only scheduled on a node that has at least that much available.

Limits are the maximum the container is allowed to use. CPU is throttled if exceeded; memory causes the container to be OOM-killed if exceeded.

Without requests/limits: pods compete for all available resources on a node (noisy-neighbour problem), the scheduler has no basis for good placement decisions, and one greedy pod can starve all others on the same node.

resources:
  requests: { cpu: 100m, memory: 128Mi }   # guaranteed minimum
  limits:   { cpu: 500m, memory: 512Mi }   # hard ceiling

# QoS classes determined by requests/limits:
# Guaranteed: requests == limits  → highest priority, last to be evicted
# Burstable:  requests < limits   → medium priority
# BestEffort: no requests/limits  → first to be evicted under pressure

Kubernetes
Hard
Explain the Kubernetes control loop / reconciliation pattern and how it relates to controllers.
▼

Kubernetes is fundamentally a reconciliation system. Every controller runs a loop:

1
Observe: watch the API server for changes to the desired state (spec)
2
Diff: compare desired state with current state
3
Act: take actions to make current state match desired state
4
Repeat: controllers are idempotent — safe to run the loop repeatedly

Example: the Deployment controller watches Deployments. If you change replicas: 3 to replicas: 5, the controller creates 2 new pods. If 2 pods die, the controller creates 2 replacements. It never stops watching.

This pattern is why Kubernetes is self-healing: desired state is always the source of truth, and controllers constantly drive actual state toward it. Custom controllers (Operators) extend this to any domain-specific resource.

Kubernetes
Hard
How does the Kubernetes scheduler decide which node to place a pod on?
▼

Scheduling happens in two phases:

1. Filtering (Predicates): Eliminate nodes that cannot run the pod:

  • Node has enough CPU/memory for the pod's requests
  • Node satisfies nodeSelector and nodeAffinity rules
  • No taints on the node that the pod doesn't tolerate
  • Pod's port requirements don't conflict with existing pods

2. Scoring (Priorities): Rank remaining nodes:

  • Least-requested resources (spread load)
  • Node affinity/anti-affinity preferences
  • Pod topology spread constraints
  • Image locality (node already has the image)

The pod is assigned to the highest-scoring node. The scheduler writes the binding back to the API server, and the node's kubelet picks it up and starts the container.

Kubernetes
Hard
What is the difference between Deployment, StatefulSet, and DaemonSet? Give a production use case for each.
▼

Deployment — stateless, interchangeable pods. Any pod can serve any request. Rolling updates create new pods before terminating old ones. Use case: REST API, frontend app, worker processes.

StatefulSet — pods have stable identities: predictable names (postgres-0, postgres-1), stable network IDs, and stable storage via volumeClaimTemplates. Pods start/stop in order. Use case: PostgreSQL primary/replica, Kafka, Elasticsearch, Redis cluster.

DaemonSet — runs exactly one pod on every node (or a subset). When new nodes join the cluster, a new pod is automatically scheduled on them. Use case: Fluentd log collector, Datadog agent, network plugin (Calico), Falco security scanner.

kubectl get pods -n monitoring  # DaemonSet pods — one per node
kubectl get pods -n db          # StatefulSet: postgres-0, postgres-1, postgres-2
Scenario-based questions
Scenario
Hard
Your deployment rolls out a new image and traffic starts returning 500 errors. How do you respond?
▼
1
Immediate rollback — don't investigate the new code while users are impacted
2
Diagnose — check logs, events, resource metrics on the bad revision
3
Fix and re-deploy — after rollback and root cause identified
# 1. Immediate rollback
kubectl rollout undo deployment/api

# 2. Check rollout history
kubectl rollout history deployment/api
kubectl rollout history deployment/api --revision=5

# 3. Inspect the bad revision's logs and events
kubectl logs -l app=api --since=10m --timestamps
kubectl describe pods -l app=api
kubectl get events --sort-by='.lastTimestamp' -n production

# Rollback to specific revision
kubectl rollout undo deployment/api --to-revision=3

Prevention: Implement a readinessProbe so pods are only added to the load balancer once they pass health checks, preventing bad pods from ever receiving traffic during rollout. Use canary deployments (Argo Rollouts / Flagger) for gradual traffic shifting.

A pod is stuck in Pending state. Walk through your debugging process.
▼

Pending means the scheduler has not yet assigned the pod to a node. The first command is always:

kubectl describe pod <pod-name> -n <namespace>
# Look at Events section at the bottom

Common causes and fixes:

1. Insufficient resources — 0/3 nodes are available: 3 Insufficient cpu

Troubleshooting
Troubleshoot
Medium
A pod shows OOMKilled status repeatedly. What is happening and how do you fix it?
▼

OOMKilled (Out Of Memory Killed) means the container exceeded its memory limit and the Linux kernel's OOM killer terminated the process. Kubernetes then restarts the container, causing a restart loop.

kubectl describe pod <pod> | grep -A3 "Last State"
# Last State: Terminated  Reason: OOMKilled  Exit Code: 137

kubectl top pod <pod> --containers  # current memory usage

Diagnosis:

  • Memory limit is set too low for the workload — increase it
  • Memory leak in the application — profile and fix the code
  • Large in-memory dataset — consider moving to external cache/DB

Fix: Increase the memory limit to 20% above the observed peak. Add a memory metric to your HPA or alert on memory pressure. Use VPA (Vertical Pod Autoscaler) in recommendation mode to observe the correct value before applying.

Troubleshoot
Hard
Two services in different namespaces cannot communicate. Walk through how you debug and fix this.
▼ 
# 1. Verify services exist and have endpoints
kubectl get svc -n ns-a
kubectl get endpoints -n ns-a api-svc
# No endpoints = no matching pods (check label selector)

# 2. Test DNS from inside a pod
kubectl exec -it debug-pod -n ns-b -- \
  nslookup api-svc.ns-a.svc.cluster.local

# 3. Test connectivity
kubectl exec -it debug-pod -n ns-b -- \
  curl -v http://api-svc.ns-a.svc.cluster.local:3000/health

# 4. Check NetworkPolicy — default deny blocks cross-namespace
kubectl get networkpolicy -n ns-a -o yaml

NetworkPolicy fix — allow ingress from namespace ns-b:

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: allow-from-ns-b, namespace: ns-a }
spec:
  podSelector: { matchLabels: { app: api } }
  ingress:
  - from:
    - namespaceSelector:
        matchLabels: { kubernetes.io/metadata.name: ns-b }
    ports: [{ protocol: TCP, port: 3000 }]
Troubleshoot
Hard
Your pods are being evicted frequently during peak load. What causes this and how do you prevent it?
▼

Eviction occurs when a node runs low on resources. kubelet evicts BestEffort pods first, then Burstable, never Guaranteed. The root causes are:

  • Memory pressure — node's actual memory use exceeds eviction threshold (default 100Mi free)
  • Disk pressure — ephemeral storage or image storage fills up
  • CPU throttling is not eviction — excess CPU is throttled, not evicted

Prevention:

# 1. Set requests == limits to get Guaranteed QoS (never evicted first)
resources:
  requests: { cpu: 500m, memory: 512Mi }
  limits:   { cpu: 500m, memory: 512Mi }

# 2. Set ephemeral-storage limits to prevent disk pressure eviction
ephemeral-storage: 1Gi

# 3. Configure Cluster Autoscaler to add nodes before node memory fills
# 4. Monitor with PodDisruptionBudget + alerts on eviction events
kubectl get events -A --field-selector reason=Evicted
Troubleshoot
Medium
A container exits immediately after starting with exit code 1. How do you debug it without being able to exec into it?
▼ 
# 1. Read logs of the crashed container
kubectl logs <pod> --previous             # logs of last crash
kubectl logs <pod> -c <container> --previous

# 2. Get exit code and reason
kubectl describe pod <pod>
# Look for: Last State → Terminated → Reason + Exit Code

# 3. Override the entrypoint to debug interactively
kubectl run debug --image=my-app:v2 \
  --restart=Never \
  --rm -it \
  -- /bin/sh    # override CMD and stay alive

# 4. If the image has no shell, use an ephemeral debug container (K8s 1.23+)
kubectl debug -it <pod> --image=busybox --target=<container>

# Common exit codes:
# 1   = application error (check logs)
# 127 = command not found (wrong ENTRYPOINT path)
# 137 = SIGKILL / OOMKilled (memory limit)
# 143 = SIGTERM not handled (graceful shutdown issue)
