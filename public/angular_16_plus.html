What changed in Angular 16, 17, and 18

Angular went through its most significant evolution in years across these three versions. The core team introduced a brand-new reactivity model (Signals), made NgModules optional (Standalone), overhauled the template syntax, and began the journey toward removing Zone.js dependency entirely.

What are Angular Signals?

A Signal is a reactive value wrapper that knows its current value and notifies interested consumers when that value changes. Unlike a plain TypeScript variable, a signal is "reactive" — any template, computed(), or effect() that reads a signal is automatically re-evaluated when the signal changes.

Before Signals, Angular relied on Zone.js to detect when something had happened and then checked every component in the tree. Signals flip this model: instead of Angular asking "did anything change?", signals tell Angular exactly what changed and which templates depend on it.

Without signals: Zone.js triggers a full tree check. With signals: only the component that consumed the changed signal re-renders.

signal() — creating a WritableSignal

You create a signal by calling signal(initialValue). The return type is WritableSignal<T>. To read the value you call it as a function: count(). To change the value you use one of three mutation methods.

counter.component.ts
import { Component, signal } from '@angular/core';

@Component({
  standalone: true,
  template: `
    <p>Count is: {{ count() }}</p>
    <button (click)="increment()">+1</button>
    <button (click)="decrement()">-1</button>
    <button (click)="reset()">Reset</button>
  `
})
export class CounterComponent {
  // signal() creates a WritableSignal<number> with initial value 0
  count = signal(0);

  increment() { this.count.update(v => v + 1); }  // current value → new value
  decrement() { this.count.update(v => v - 1); }
  reset()     { this.count.set(0); }              // direct replacement
}

Three ways to update a signal

MethodSignatureWhen to useExample
.set(value)(newVal: T) => voidYou already have the new valuename.set('Alice')
.update(fn)((current: T) => T) => voidNew value derived from currentcount.update(v => v + 1)
.mutate(fn)((val: T) => void) => voidRemoved in v17. Used to mutate objects in-place—
Angular 17 Breaking Change — .mutate() removed

.mutate() was removed in Angular 17 because mutation without creating a new reference breaks the contract that signals use for equality checking. Use .update() with an immutable pattern: items.update(arr => [...arr, newItem])

Signal equality — how Angular decides if a re-render is needed

By default, Angular uses Object.is() (strict reference equality) to compare the old and new signal value. If you pass the same primitive value, no re-render fires. For complex objects, you can supply a custom equality function:

import _ from 'lodash';

// Deep equality — signal won't notify if the new array has the same contents
const data = signal(['a', 'b'], { equal: _.isEqual });
data.set(['a', 'b']);  // different array reference but equal content → no notification

// Making a signal read-only for public API
private _count = signal(0);
readonly count = this._count.asReadonly();  // returns Signal<number>, no .set()/.update()

computed() — memoised derived state

computed() creates a read-only, memoised signal whose value is derived from one or more other signals. It is lazy and memoised — it only recalculates when a dependency signal has changed AND something reads it. Multiple reads return the cached result.

The key difference from a getter: a getter recalculates on every access. A computed signal tracks its dependencies and skips recalculation if none of them changed.

cart.service.ts
import { Injectable, signal, computed } from '@angular/core';

@Injectable({ providedIn: 'root' })
export class CartService {
  private _items = signal<{ name: string; price: number; qty: number }[]>([]);
  readonly items = this._items.asReadonly();

  // computed() reads _items() — tracked as a dependency
  readonly itemCount = computed(() => this._items().length);
  readonly subtotal  = computed(() =>
    this._items().reduce((sum, i) => sum + i.price * i.qty, 0)
  );
  // computed() can read other computed() — chain of dependencies
  readonly tax   = computed(() => this.subtotal() * 0.1);
  readonly total = computed(() => this.subtotal() + this.tax());

  add(item: { name: string; price: number; qty: number }) {
    this._items.update(arr => [...arr, item]);
  }
  remove(name: string) {
    this._items.update(arr => arr.filter(i => i.name !== name));
  }
}
How computed() tracks dependencies

When a computed() runs for the first time, Angular records every signal it calls (signal()). Those become its dependencies. Next time any of those signals change, Angular marks the computed as stale. The computation does NOT re-run immediately — it runs lazily only when something reads the computed value again.

effect() — running side effects reactively

effect() creates a reactive side effect — a function that runs once immediately on creation, then re-runs whenever any signal it reads changes. It must be called inside an injection context (constructor or field initializer).

theme.component.ts
import { Component, signal, effect, inject, Injector, runInInjectionContext } from '@angular/core';

@Component({ standalone: true })
export class ThemeComponent {
  theme = signal<'light' | 'dark'>('light');

  constructor() {
    // effect runs once now, then again every time theme() changes
    effect(() => {
      // Angular tracks that this effect reads theme()
      document.body.setAttribute('data-theme', this.theme());
      localStorage.setItem('theme', this.theme());
    });
  }
}

// If you must call effect() outside the constructor, use runInInjectionContext
export class SomeClass {
  private injector = inject(Injector);

  ngOnInit() {
    runInInjectionContext(this.injector, () => {
      effect(() => console.log(this.theme()));
    });
  }
}
Anti-pattern: writing a signal inside effect()

Setting a signal inside effect() creates circular data flow and potential infinite loops. Angular will warn about this in development. If you need a value derived from another signal, always use computed().

Wrong — signal write in effect
// Anti-pattern: effect sets a signal
effect(() => {
  this.fullName.set(
    `${this.first()} ${this.last()}`
  );
});
Correct — use computed()
// Derived state belongs in computed()
fullName = computed(() =>
  `${this.first()} ${this.last()}`
);

Cleanup inside effect()

You can register a cleanup function inside an effect using onCleanup(). The cleanup runs before the effect re-runs (when its dependencies changed) and when the effect is destroyed.

effect(onCleanup => {
  const subscription = someObservable.subscribe(v => console.log(v));
  onCleanup(() => subscription.unsubscribe());  // runs before next execution
});

Signal inputs and outputs

Signal inputs — replacing @Input()

Angular 17.1 introduced signal inputs as a stable replacement for @Input(). Instead of a plain property, signal inputs are Signal<T> values — you read them with this.title(). This means any computed() that reads a signal input automatically recalculates when the parent passes a new value — no need for ngOnChanges.

user-card.component.ts
import { Component, input, output, computed } from '@angular/core';

@Component({
  standalone: true,
  selector: 'user-card',
  template: `
    <h2>{{ displayName() }}</h2>
    <p>{{ email() }}</p>
    @if (isAdmin()) { <span class="badge">Admin</span> }
    <button (click)="deleteClicked.emit(id())">Delete</button>
  `
})
export class UserCardComponent {
  // required input — NG error if parent doesn't supply it
  id    = input.required<string>();
  name  = input.required<string>();
  email = input.required<string>();

  // optional input with default value
  role  = input<'user' | 'admin'>('user');

  // alias — parent uses [label]="..." but class property is title
  title = input<string>('', { alias: 'label' });

  // signal output replaces @Output() clicked = new EventEmitter<string>()
  deleteClicked = output<string>();

  // computed() from signal inputs — auto-updates when name() or role() changes
  displayName = computed(() => this.name().toUpperCase());
  isAdmin     = computed(() => this.role() === 'admin');
}

If input.required() is not supplied by the parent, Angular throws at runtime:

Error: NG0950: Required input 'name' from component UserCardComponent must be specified.

This is stronger than the old @Input() required name!: string TypeScript assertion, which was never enforced at runtime by Angular.

toSignal() and toObservable() — RxJS interop

toSignal() — consume an Observable as a Signal

toSignal() subscribes to an Observable and exposes the latest emitted value as a Signal. The subscription is automatically cleaned up when the injection context is destroyed — no manual unsubscription needed.

users.component.ts
import { Component, inject, computed } from '@angular/core';
import { toSignal } from '@angular/core/rxjs-interop';
import { UserService } from './user.service';

@Component({
  standalone: true,
  template: `
    @if (users().length) {
      @for (u of users(); track u.id) {
        <p>{{ u.name }}</p>
      }
    } @else { <p>No users found.</p> }
  `
})
export class UsersComponent {
  private svc = inject(UserService);

  // initialValue sets the signal value before first emission
  // Without it, type would be User[] | undefined
  users = toSignal(this.svc.getAll$(), { initialValue: [] as User[] });

  adminCount = computed(() => this.users().filter(u => u.role === 'admin').length);
}
Common bug: toSignal() fires HTTP twice

Calling toSignal(this.http.get(...)) directly subscribes to a cold observable. If the component mounts twice or the service has multiple injections, two HTTP requests fire. Always add shareReplay(1) to share the observable before converting:

// service.ts — share before converting
private readonly users$ = this.http.get<User[]>('/api/users').pipe(shareReplay(1));
readonly users = toSignal(this.users$, { initialValue: [] });
// HTTP fires exactly once — all toSignal() consumers share the same subscription

toObservable() — feed a Signal into an RxJS pipeline

search.component.ts
import { toSignal, toObservable } from '@angular/core/rxjs-interop';

export class SearchComponent {
  searchTerm = signal('');

  // Convert signal → observable to use debounce + cancellation
  results = toSignal(
    toObservable(this.searchTerm).pipe(
      debounceTime(300),
      distinctUntilChanged(),
      switchMap(term => this.searchSvc.search(term))  // cancels in-flight
    ),
    { initialValue: [] }
  );
}

linkedSignal — Angular 18

linkedSignal is a new primitive in Angular 18 that creates a writable signal whose value is reset/synchronised whenever a source signal changes. It solves a common pattern where you want a local writable signal that "resets" when an input changes.

tabs.component.ts — Angular 18
import { Component, input, linkedSignal } from '@angular/core';

@Component({ standalone: true })
export class TabsComponent {
  tabs = input.required<string[]>();

  // activeTab is writable (user can change it) but resets to tabs()[0]
  // whenever the tabs input itself changes
  activeTab = linkedSignal(() => this.tabs()[0]);

  selectTab(tab: string) {
    this.activeTab.set(tab);  // user changes — behaves as writable signal
  }
  // When parent changes [tabs], activeTab resets to the new tabs()[0] automatically
}

Standalone components — no NgModule needed

Standalone components, directives, and pipes were stabilised in Angular 15. With Angular 17 they became the default for all new projects. A standalone component is self-contained — it declares its own dependencies in an imports array rather than relying on an NgModule to provide them.

NgModule pattern: components declare which module owns them. Standalone pattern: each component owns its own imports.
user-list.component.ts
import { Component, inject, signal, computed } from '@angular/core';
import { NgOptimizedImage } from '@angular/common';
import { RouterLink } from '@angular/router';
import { UserCardComponent } from './user-card.component';

@Component({
  standalone: true,                        // ← marks it standalone
  selector: 'user-list',
  imports: [NgOptimizedImage, RouterLink, UserCardComponent],  // ← self-managed
  template: `
    @for (user of users(); track user.id) {
      <user-card [name]="user.name" [email]="user.email" />
    }
  `
})
export class UserListComponent {
  private svc = inject(UserService);
  users = toSignal(this.svc.getAll$(), { initialValue: [] });
}

bootstrapApplication() and provide*() functions

main.ts — fully standalone, no AppModule
import { bootstrapApplication } from '@angular/platform-browser';
import { provideRouter, withComponentInputBinding, withPreloading, PreloadAllModules } from '@angular/router';
import { provideHttpClient, withInterceptorsFromDi } from '@angular/common/http';
import { provideAnimationsAsync } from '@angular/platform-browser/animations/async';
import { provideClientHydration } from '@angular/platform-browser';

bootstrapApplication(AppComponent, {
  providers: [
    provideRouter(routes,
      withComponentInputBinding(),       // route params → @Input / input()
      withPreloading(PreloadAllModules)
    ),
    provideHttpClient(withInterceptorsFromDi()),
    provideAnimationsAsync(),
    provideClientHydration(),              // SSR hydration (v17+)
  ]
});

Why provide*() functions instead of forRoot()?

provide*() functions are tree-shakeable provider factories. Because they are plain functions, unused features are removed by the bundler at build time. RouterModule.forRoot() always included the entire router feature set. provideRouter(routes, withPreloading(...)) only includes the features you explicitly add.

Old (forRoot)New (provide*)
RouterModule.forRoot(routes)provideRouter(routes)
HttpClientModuleprovideHttpClient()
BrowserAnimationsModuleprovideAnimationsAsync()

Lazy routes with loadComponent

app.routes.ts
import { Routes } from '@angular/router';

export const routes: Routes = [
  {
    path: 'dashboard',
    // Lazy-load a single standalone component — no wrapper module needed
    loadComponent: () => import('./dashboard/dashboard.component')
      .then(m => m.DashboardComponent)
  },
  {
    path: 'admin',
    // Route-scoped providers — services scoped to this subtree only
    providers: [AdminService, { provide: ADMIN_LEVEL, useValue: 'superadmin' }],
    loadChildren: () => import('./admin/admin.routes')
  },
  {
    path: 'user/:id',
    loadComponent: () => import('./user/user.component').then(m => m.UserComponent),
    resolve: { profile: userProfileResolver }
  }
];

// user.component.ts — route params auto-bound (requires withComponentInputBinding)
export class UserComponent {
  id      = input.required<string>();   // from :id route param
  tab     = input('overview');          // from ?tab= query param
  profile = input<UserProfile | null>(null);  // from resolve.profile
  // No ActivatedRoute needed!
}

The inject() function

The inject() function is a functional alternative to constructor injection. Angular 16 made it possible to call inject() in field initializers — properties declared directly on the class body. This eliminates the need for a constructor in most components.

Constructor injection (old)
@Component({ standalone: true })
export class OldComp {
  users: User[] = [];

  constructor(
    private svc: UserService,
    private router: Router
  ) {}

  ngOnInit() {
    this.svc.getAll$().subscribe(
      u => this.users = u
    );
  }
}
inject() at field level (new)
@Component({ standalone: true })
export class NewComp {
  private svc    = inject(UserService);
  private router = inject(Router);
  private ref    = inject(DestroyRef);

  users = toSignal(
    this.svc.getAll$(),
    { initialValue: [] }
  );
  // No constructor, no ngOnInit
}
Injection context — when can you call inject()?
  • Inside a class constructor
  • In a field initializer (evaluated during construction)
  • Inside a factory function for a provider
  • Inside runInInjectionContext(injector, fn)
  • Inside a functional guard, resolver, or interceptor (they receive an injection context)

Calling inject() in ngOnInit, event handlers, Promises, or setTimeout throws:

NG0203: inject() must be called from an injection context such as a constructor, a factory function, a field initializer, or a function used with runInInjectionContext.

DestroyRef and takeUntilDestroyed()

DestroyRef is an injectable that lets you register cleanup callbacks without implementing OnDestroy. takeUntilDestroyed() is a pipeable RxJS operator that automatically completes an Observable when the current injection context is destroyed.

realtime.component.ts
import { Component, inject, DestroyRef } from '@angular/core';
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';

@Component({ standalone: true })
export class RealtimeComponent {
  private wsService  = inject(WebSocketService);
  private destroyRef = inject(DestroyRef);

  messages = signal<string[]>([]);

  constructor() {
    // Automatically unsubscribes when component is destroyed
    this.wsService.messages$.pipe(
      takeUntilDestroyed(this.destroyRef)
    ).subscribe(msg => {
      this.messages.update(arr => [...arr, msg]);
    });

    // Register any additional cleanup
    this.destroyRef.onDestroy(() => {
      console.log('RealtimeComponent destroyed — cleaning up');
    });
  }
}

// Shorthand: takeUntilDestroyed() with no arg auto-injects DestroyRef
// Only works in field initializer or constructor (injection context)
export class CleanComponent {
  data$ = inject(DataService).stream$.pipe(takeUntilDestroyed());
}

InjectionToken — type-safe non-class DI

Use InjectionToken when you want to inject a value that is not a class — configuration objects, environment variables, feature flags, primitive values, or abstract interfaces.

tokens.ts
import { InjectionToken } from '@angular/core';

export interface AppConfig {
  apiUrl: string;
  maxRetries: number;
  features: { darkMode: boolean; beta: boolean };
}

// providedIn + factory = tree-shakeable, no NgModule, no explicit register
export const APP_CONFIG = new InjectionToken<AppConfig>('app.config', {
  providedIn: 'root',
  factory: () => ({
    apiUrl: 'https://api.prod.financesphere.io',
    maxRetries: 3,
    features: { darkMode: true, beta: false }
  })
});

// consume it anywhere with full type safety
@Injectable({ providedIn: 'root' })
export class ApiService {
  private cfg = inject(APP_CONFIG);  // type is AppConfig
  private base = this.cfg.apiUrl;

  get<T>(path: string) {
    return this.http.get<T>(`${this.base}${path}`);
  }
}

Route-level providers — replacing lazy module scoping

In NgModule-based apps, lazy-loaded modules created their own injector that scoped services to that module's lifetime. In standalone Angular, you replicate this with the providers array on a route definition. Services in route-level providers are shared by all components in that route subtree and are destroyed when the user navigates away.

app.routes.ts
{
  path: 'admin',
  // EnvironmentProviders scoped to /admin subtree — created on nav in, destroyed on nav out
  providers: [
    AdminUserService,
    AuditLogService,
    { provide: AUTH_LEVEL, useValue: 'admin' }
  ],
  loadChildren: () => import('./admin/admin.routes')
}

// vs component providers — each component instance gets its own service
@Component({
  providers: [WizardStateService]  // destroyed with the component — one per instance
})

Change Detection — Default strategy

Angular's default change detection strategy uses Zone.js. Zone.js monkey-patches all browser async APIs (setTimeout, Promise.then, XHR, event listeners) and notifies Angular whenever any of them complete. Angular then runs a top-down check of the entire component tree to find and update any bindings that changed.

This works, but it is inefficient for large apps: every click, every HTTP response, every timer tick checks every component — even those with no data relevant to the change.

Change Detection — OnPush strategy

With ChangeDetectionStrategy.OnPush, Angular marks the component and its entire subtree as "clean" and skips them during change detection unless one of these specific triggers occurs:

  • @Input() or input() reference changes — the parent passes a new object/array reference (not a mutation)
  • An event in the component fires — click, input, keydown within the component's template
  • async pipe emits — an Observable connected via | async emits a new value
  • Signal read in template changes (v17+) — a signal whose value changed since last render
  • ChangeDetectorRef.markForCheck() — manual trigger (escape hatch)
product-list.component.ts
import { Component, ChangeDetectionStrategy, input, computed } from '@angular/core';

@Component({
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <p>Showing {{ visibleCount() }} of {{ products().length }} products</p>
    @for (p of visibleProducts(); track p.id) {
      <product-card [product]="p" />
    }
  `
})
export class ProductListComponent {
  products = input<Product[]>([]);
  filter   = input('');
  maxShow  = input(20);

  visibleProducts = computed(() =>
    this.products()
      .filter(p => p.name.toLowerCase().includes(this.filter().toLowerCase()))
      .slice(0, this.maxShow())
  );
  visibleCount = computed(() => this.visibleProducts().length);
}

Local Change Detection with Signals (Angular 17)

Angular 17 introduced Local Change Detection — a significant performance improvement. When a signal used in a component's template changes, Angular now marks only that specific component as dirty, not its ancestors. For ancestors, it sets a "has child views to refresh" flag so the next change detection pass knows it needs to traverse into those children without checking the ancestors themselves.

This requires two conditions: the component must use OnPush and the binding must use Signals. When both are true, a signal change causes the minimum possible amount of change detection work.

Pre-v17: signal change marks all ancestors dirty. v17+: ancestors only get a traversal flag, not a dirty mark. Only the signal consumer re-renders.

Zoneless Change Detection — Angular 18

Angular 18 introduced experimental support for running entirely without Zone.js. In zoneless mode, Angular does not rely on Zone.js to know when to check for changes. Instead, it only re-renders components when a signal they consume changes, when ChangeDetectorRef.markForCheck() is called, or when an event fires inside the component.

main.ts — enable zoneless (experimental)
bootstrapApplication(AppComponent, {
  providers: [
    provideExperimentalZonelessChangeDetection()
  ]
});

// Also remove zone.js from angular.json polyfills:
// "polyfills": ["zone.js"]  →  "polyfills": []
Zoneless is experimental in Angular 18

Third-party libraries that rely on Zone.js to trigger change detection (e.g. they call setTimeout or emit events without using Signals or markForCheck()) will stop working correctly in zoneless mode. Test every dependency before adopting this in production.

New built-in control flow — @if, @for, @switch

Angular 17 replaced structural directives with a new built-in template syntax. The new syntax is compiled directly into the template engine — no import needed, better type narrowing, better performance, and clearer developer error messages.

*ngIf / *ngFor / ngSwitch
<div *ngIf="user; else noUser">
  {{ user.name }}
</div>
<ng-template #noUser>
  Loading...
</ng-template>

<li *ngFor="let item of items;
           trackBy:trackById;
           let i = index">
  {{ i }}: {{ item.name }}
</li>
@if / @for / @switch
@if (user) {
  <div>{{ user.name }}</div>
} @else if (isLoading) {
  <spinner />
} @else {
  <p>Not found.</p>
}

@for (item of items; track item.id;
      let i = $index) {
  <li>{{ i }}: {{ item.name }}</li>
} @empty {
  <li>No items.</li>
}

@for requires track — not optional

Unlike trackBy in *ngFor which was optional, track in @for is mandatory. Omitting it is a compile error. Using track $index is allowed but disables Angular's list reconciliation — it destroys and recreates every DOM element on re-render. Always use a unique identifier: track item.id.

@defer blocks — declarative lazy loading

@defer is a template-level feature that downloads a component's JavaScript bundle only when a trigger condition is met. Before the trigger fires, Angular shows the @placeholder. While the bundle is downloading, it shows @loading. If the download fails, it shows @error.

<h2>Product Details</h2>
<!-- Reviews section is below the fold — load it lazily -->

@defer (on viewport; prefetch on idle) {
  <reviews-section [productId]="id()" />

} @placeholder (minimum 300ms) {
  <!-- Shown until trigger fires — min prevents flash -->
  <div class="reviews-skeleton" aria-busy="true"></div>

} @loading (minimum 200ms; after 100ms) {
  <!-- Shown while bundle downloads (after 100ms delay) -->
  <loading-spinner />

} @error {
  <p>Reviews could not load. <a (click)="retry()">Try again</a></p>
}

Best practice: combine on viewport + prefetch on idle

The prefetch option downloads the bundle in the background before the trigger fires. Combining on viewport; prefetch on idle means: download the bundle during browser idle time, but only render the component when it enters the viewport. The user gets an instant render because the bundle is already cached locally.

Interview Q&A — Signals

These questions reflect what a developer with 3–5 years of Angular experience is expected to know. Each answer explains the mechanism deeply, not just the surface API.

SignalsMedium
What is the difference between signal(), computed(), and effect()? When would you use each one?
▼

signal() — mutable source of truth

Use signal() for state that your component or service owns and mutates directly. It is the raw, writable reactive value. Every other reactive primitive derives from or reacts to signals.

isLoading    = signal(false);
selectedTab  = signal<'overview' | 'details'>('overview');
currentUser  = signal<User | null>(null);

computed() — read-only derived state, lazily memoised

Use computed() whenever a value can be derived from other signals. The key properties: (1) lazy — only re-evaluates when a dependency changed AND something reads it. (2) Memoised — multiple reads return the same cached result until a dependency changes. (3) Read-only — you cannot call .set() on it.

Never store derived data in a writable signal that you manually sync — that creates bugs and race conditions. Let computed() own all derived state.

fullName    = computed(() => `${this.firstName()} ${this.lastName()}`);
isFormValid = computed(() => !!this.email() && this.password().length >= 8);
admins      = computed(() => this.users().filter(u => u.role === 'admin'));

effect() — side effects outside Angular's reactivity

Use effect() only for things that escape Angular's reactivity: writing to DOM attributes, localStorage, sending analytics, logging, calling third-party libraries. It runs once on creation, then re-runs whenever any signal it reads changes. It must be in an injection context.

Never use effect() to derive new signal state — that belongs in computed().

effect(() => {
  document.title = `${this.pageName()} | My App`;          // DOM side effect
  analytics.page(this.pageName());                          // analytics
  localStorage.setItem('lastPage', this.pageName());        // storage
});
SignalsHard
A computed() that reads an array signal stops updating when items are pushed into the array. Why does this happen and what are the solutions?
▼

Root cause: Angular uses Object.is() for equality

By default, signals compare old and new values using Object.is() — strict reference equality. When you call this.items().push(newItem), the array is mutated in place. The array reference before and after the push is the same object. Object.is(arr, arr) returns true, so Angular considers the signal unchanged and does not notify any computed() or template that depends on it.

// BUG — mutation in place
items = signal<string[]>([]);
computed_count = computed(() => this.items().length);  // never updates!

addItem(item: string) {
  this.items().push(item);    // same reference — signal sees no change
}

Solution 1: use .update() with an immutable pattern (recommended)

addItem(item: string) {
  this.items.update(arr => [...arr, item]);   // new array reference
}
removeItem(idx: number) {
  this.items.update(arr => arr.filter((_, i) => i !== idx));
}
updateItem(idx: number, val: string) {
  this.items.update(arr => arr.map((v, i) => i === idx ? val : v));
}

Solution 2: custom equality function for deep comparison

import _ from 'lodash';

// Signal won't notify if new array has same contents (deep equal)
items = signal<string[]>([], { equal: _.isEqual });

// Now mutation is detectable — but only if you call .set() with new value
addItem(item: string) {
  this.items.set([...this.items(), item]);
}

Why immutable updates are preferred in Angular signals

Immutable updates (creating a new array/object instead of mutating) make change detection predictable — Angular can always tell what changed by reference comparison alone. They also enable features like undo/redo, time-travel debugging, and straightforward memoization.

SignalsHard
How does toSignal() manage its subscription lifecycle? What happens if you call it outside an injection context?
▼

Lifecycle management via DestroyRef

toSignal() internally calls inject(DestroyRef) when it runs. It registers an onDestroy callback that unsubscribes from the Observable when the current injection context (the component or service) is destroyed. This is why you do not need to manually unsubscribe — the cleanup is automatic.

What happens outside injection context

Calling toSignal() in ngOnInit, an event handler, or a Promise throws:

NG0203: toSignal() can only be used within an injection context such as a constructor, a factory function, a field initializer, or a function used with runInInjectionContext.

How to use toSignal() outside injection context

export class SearchComponent implements OnInit {
  private injector = inject(Injector);
  results = signal<Result[]>([]);

  ngOnInit() {
    // Pass the injector explicitly to toSignal()
    const results = toSignal(this.searchService.results$, {
      initialValue: [],
      injector: this.injector    // manually provide injection context
    });
    this.results = results;
  }
}

manualCleanup option

// If you want to manage the subscription yourself:
const signal = toSignal(observable$, { manualCleanup: true });
// Signal will NOT be automatically cleaned up — your responsibility
SignalsHard
What is linkedSignal in Angular 18? What problem does it solve that computed() cannot?
▼

The problem computed() cannot solve

computed() is read-only. It is perfect for values that are always derived from other signals. But sometimes you need a value that: (1) starts derived from a source signal, (2) can be overridden by the user (written to), AND (3) resets back to the derived value when the source changes.

Classic example: a selected tab that resets to the first tab whenever the tab list changes, but allows user selection in between.

Without linkedSignal — messy workaround with effect()

// BAD: using effect() to sync signals
tabs = input.required<string[]>();
activeTab = signal('');   // writable

constructor() {
  // Anti-pattern: effect writes to a signal
  effect(() => { this.activeTab.set(this.tabs()[0]); });
}

With linkedSignal — clean and correct

tabs = input.required<string[]>();

// linkedSignal: writable, but resets when tabs() changes
activeTab = linkedSignal(() => this.tabs()[0]);

selectTab(tab: string) {
  this.activeTab.set(tab);    // user can override freely
}
// When parent changes [tabs]="newTabs", activeTab auto-resets to newTabs[0]

Advanced form with computation

// linkedSignal with source + computation pattern (Angular 18 extended form)
activeTab = linkedSignal({
  source: this.tabs,                      // watch this signal
  computation: (tabs, previous) => {
    // If previous selection still exists in new tabs, keep it
    if (previous?.value && tabs.includes(previous.value)) return previous.value;
    return tabs[0];  // otherwise reset to first
  }
});

Interview Q&A — Change Detection

Change DetectionHard
Explain exactly how Local Change Detection in Angular 17 works with OnPush and Signals. Why do you need both?
▼

Background: how OnPush worked before v17

When a signal in a child component changed before v17, Angular used markViewDirty(), which marked the component AND all its ancestors as dirty. This triggered a full change detection pass from the root down to the changed component — checking every ancestor along the way.

What Angular 17 changed: markAncestorsForTraversal

Angular 17 introduced a new function markAncestorsForTraversal() that replaces markViewDirty() for signal-driven updates. Instead of marking ancestors as dirty, it marks them as "has child views to refresh" (HasChildViewsToRefresh flag). The difference:

  • Dirty = check this component's bindings
  • HasChildViewsToRefresh = traverse into children but skip checking this component's own bindings

Result: only the component that actually consumed the changed signal has its template re-evaluated. Ancestors are traversed but not checked.

Why you need BOTH OnPush AND Signals

  • OnPush without signals: still uses Zone.js for non-signal updates. Zone.js triggers full tree check. Local CD not active.
  • Signals without OnPush (Default strategy): Zone.js still triggers full tree check after every event. The signal change also triggers a re-render but the "local" benefit is masked.
  • OnPush + Signals: Zone.js still runs but can skip the component (OnPush + no dirty inputs). Signal change uses markAncestorsForTraversal() — only the consumer is checked. Maximum performance.
@Component({
  changeDetection: ChangeDetectionStrategy.OnPush,  // required condition 1
  template: `<p>{{ count() }}</p>`                  // signal in template = required condition 2
})
export class OptimisedComponent {
  count = signal(0);  // when count.set(1) fires, only THIS component re-renders
}
Change DetectionHard
A third-party charting library updates its internal state outside Angular's zone, and the chart never re-renders. What are your options from worst to best?
▼

Why it happens

Zone.js intercepts standard browser async APIs (setTimeout, Promise, XHR, addEventListener). Libraries that use requestAnimationFrame, WebGL contexts, native WebSocket, or their own internal event systems run outside Zone.js. Angular is never notified and never checks the affected components.

Option 1 (worst): remove OnPush from the component

Removing OnPush makes the component Default strategy — Zone.js checks it on every tick. This works but sacrifices all performance optimisation and is the wrong fix.

Option 2: wrap in NgZone.run()

constructor(private zone: NgZone) {}

ngOnInit() {
  this.chart.on('dataUpdated', (newData) => {
    this.zone.run(() => {         // bring execution into Angular zone
      this.chartData = newData;   // triggers change detection
    });
  });
}

Option 3: ChangeDetectorRef.detectChanges()

private cdr = inject(ChangeDetectorRef);

ngOnInit() {
  this.chart.on('dataUpdated', (newData) => {
    this.chartData = newData;
    this.cdr.detectChanges();   // force check this component + children
  });
}

Option 4 (best for new code): use a Signal

chartData = signal<ChartData | null>(null);

ngOnInit() {
  this.chart.on('dataUpdated', (newData) => {
    this.chartData.set(newData);   // Signal notifies template directly
    // No zone, no cdr needed — works with OnPush + zoneless
  });
}

The Signal approach is the cleanest: it works regardless of whether Zone.js is present, works with OnPush and zoneless, and is the pattern Angular 18's zoneless mode is built around.

Change DetectionMedium
What is the difference between ChangeDetectorRef.markForCheck() and ChangeDetectorRef.detectChanges()? When would you use each?
▼

markForCheck()

Marks the current component and all its ancestors as dirty in the change detection tree. Angular will check them on the next change detection cycle — not immediately. It does NOT synchronously run change detection.

Use it when: you are inside an async callback (Zone.js-patched), RxJS async pipe, or after external state update in an OnPush component and you want Angular to pick it up on its own next tick.

detectChanges()

Runs change detection synchronously and immediately on the current component and its children (down the tree). Does NOT check ancestors. Use it sparingly.

Use it when: you need the DOM to reflect a change before the next animation frame (e.g. before measuring an element's dimensions), or when integrating with non-Angular libraries that need the DOM updated synchronously.

// markForCheck — deferred, marks ancestors too
@Component({ changeDetection: ChangeDetectionStrategy.OnPush })
export class AsyncComponent {
  private cdr = inject(ChangeDetectorRef);
  data: string[] = [];

  ngOnInit() {
    someExternalLibrary.onUpdate(items => {
      this.data = items;
      this.cdr.markForCheck();   // Angular will re-check on next tick
    });
  }
}

// detectChanges — immediate, only children
export class MeasureComponent {
  private cdr = inject(ChangeDetectorRef);
  private el  = inject(ElementRef);

  showAndMeasure() {
    this.isVisible = true;
    this.cdr.detectChanges();                   // DOM updated synchronously
    const height = this.el.nativeElement.offsetHeight;  // can measure now
  }
}

Interview Q&A — Dependency Injection

DIMedium
Explain the difference between providedIn: 'root', providedIn: 'any', component-level providers, and route-level providers. When would you choose each?
▼

providedIn: 'root' — singleton across the whole application

The service is registered with the root environment injector. There is exactly one instance shared by every component, service, and guard in the app. It is also tree-shakeable — if nothing injects it, the bundler removes it.

Use for: AuthService, UserStore, ThemeService, LoggingService — things that must be singleton and app-wide.

@Injectable({ providedIn: 'root' })
export class AuthService { /* one instance, app-wide */ }

providedIn: 'any' — one instance per lazy boundary

Each lazy-loaded chunk (module or route) gets its own instance. Eagerly-loaded modules share one instance. Rarely used in modern Angular — route-level providers are more explicit and preferred.

Component providers — one instance per component instance

Each component instance creates its own injector and its own service instance. When the component is destroyed, its service is destroyed too. Sibling components each get their own instance — they do not share state.

Use for: form state services, wizard step services, autocomplete services where each component instance needs its own state.

@Component({
  providers: [WizardStateService]  // each WizardComponent instance gets its own
})
export class WizardComponent { /* this.svc is exclusive to this instance */ }

Route-level providers (standalone) — scoped to route subtree lifetime

Services are shared by all components in the route subtree and destroyed when navigating away. This is the standalone equivalent of lazy module scoping.

// app.routes.ts
{
  path: 'checkout',
  providers: [CheckoutSessionService],  // destroyed when leaving /checkout
  loadChildren: () => import('./checkout/checkout.routes')
}

  charge(amount: number) { return of({ success: true, transactionId: 'mock-123' }); }

// main.ts (production)

// test setup

Interview Q&A — Standalone Architecture

StandaloneHard
You are upgrading an Angular 12 app (NgModule-based) to Angular 17 with standalone. Walk through your migration strategy, decisions, and pitfalls.
▼

Step 1: upgrade Angular version by version

Never jump from Angular 12 to 17 in one step. Angular provides migration schematics and breaking change lists for each major version. Each version upgrade is: ng update @angular/core@13 @angular/cli@13, fix issues, test, commit, repeat up to v17.

Step 2: run the standalone migration schematic

ng generate @angular/core:standalone

This runs in three modes (pass --mode):

  • convert-to-standalone: adds standalone: true to all components and moves imports into the component
  • prune-ng-modules: removes NgModule files that have become empty
  • standalone-bootstrap: switches main.ts to use bootstrapApplication()

Run them in order. Compile and test after each pass.

Step 3: handle SharedModule

SharedModule is the biggest challenge. It typically exports dozens of components. After conversion, each component needs to import from the SharedModule directly. Two strategies:

  • Barrel export: keep SharedModule temporarily as a compatibility shim that re-exports all standalone components. Components import from it. Then gradually replace with direct imports.
  • Direct import: import each shared component individually where needed. More verbose initially but fully tree-shakeable.

Common pitfalls

  • forRoot() providers: libraries using MyLib.forRoot() must be replaced with their provide*() function equivalent or added to bootstrapApplication()'s providers array.
  • Lazy module providers: services providedIn a lazy module must move to route-level providers: arrays in app.routes.ts.
  • TestBed: all component tests must change from declarations: [MyComp] to imports: [MyComp].
  • CommonModule: standalone components include NgIf/NgFor by default via CommonModule in the imports array, but in v17+ you can use the new @if/@for syntax with no import needed.

Interview Q&A — Performance

PerformanceHard
Your Angular app has a 10,000 row table. Users report scroll lag. OnPush is applied. What else do you do?
▼

Why OnPush alone is not enough

OnPush reduces unnecessary change detection but does NOT reduce the number of DOM nodes. 10,000 <tr> elements still exist in the DOM — layout, paint, and scroll events are still slow because the browser must manage all of them.

Solution 1: CDK Virtual Scroll (renders only visible rows)

// app.component.ts

// template
<cdk-virtual-scroll-viewport itemSize="48" style="height: 600px; overflow-y: auto;">
  <table>
    <tr *cdkVirtualFor="let row of rows; trackBy: trackById">
      <td>{{ row.name }}</td>
      <td>{{ row.value }}</td>
    </tr>
  </table>
</cdk-virtual-scroll-viewport>
// Only ~20 rows are in the DOM at any time — rest are virtualised

Solution 2: @defer inside rows for complex cells

@for (row of rows; track row.id) {
  <tr>
    <td>{{ row.name }}</td>
    <td>
      @defer (on viewport) {
        <complex-cell [data]="row.chartData" />  <!-- only renders when visible -->
      } @placeholder { <span>...</span> }
    </td>
  </tr>
}

Solution 3: server-side pagination / infinite scroll

Load only 50 rows at a time. When user scrolls to the bottom, append the next page. Never load all 10,000 rows into the browser — the server query should also paginate.

Solution 4: ensure track uses a unique ID, not $index

If track $index is used, Angular destroys and recreates every DOM row when the data changes. With track row.id, Angular reuses DOM elements — only actually changed cells are updated.

ScenarioHard
Your product team wants a search-as-you-type feature. As the user types, results should update but NOT send an HTTP request on every keystroke. The results must cancel in-flight requests when a new query supersedes the old one. Implement this cleanly with Angular 17+ patterns.
▼

The design

Signals own the local UI state (search term, loading flag). RxJS handles the async coordination (debounce, cancellation with switchMap). toSignal() bridges RxJS output back to a Signal for clean template binding. No manual subscription management needed.

// search.component.ts
import { Component, signal, inject } from '@angular/core';
import { toSignal, toObservable } from '@angular/core/rxjs-interop';
import { debounceTime, distinctUntilChanged, switchMap, tap, catchError } from 'rxjs/operators';
import { of } from 'rxjs';

@Component({
  standalone: true,
  template: `
    <input
      [value]="searchTerm()"
      (input)="searchTerm.set($any($event.target).value)"
      placeholder="Search products..."
    />

    @if (isLoading()) { <loading-spinner /> }

    @if (error()) {
      <p class="error">{{ error() }}</p>
    }

    @for (result of results(); track result.id) {
      <product-card [product]="result" />
    } @empty {
      @if (!isLoading()) { <p>No results found.</p> }
    }
  `
})
export class SearchComponent {
  private searchService = inject(SearchService);

  searchTerm = signal('');
  isLoading  = signal(false);
  error      = signal<string | null>(null);

  results = toSignal(
    toObservable(this.searchTerm).pipe(   // signal → observable
      debounceTime(300),                  // wait 300ms after last keystroke
      distinctUntilChanged(),             // skip if same as previous
      tap(() => {
        this.isLoading.set(true);
        this.error.set(null);
      }),
      switchMap(term =>                   // cancel previous in-flight request
        term.trim()
          ? this.searchService.search(term).pipe(
              tap(() => this.isLoading.set(false)),
              catchError(err => {
                this.error.set('Search failed. Please try again.');
                this.isLoading.set(false);
                return of([]);
              })
            )
          : of([]).pipe(tap(() => this.isLoading.set(false)))
      )
    ),
    { initialValue: [] }
  );
}

    step4: new FormGroup({ payment: new FormControl('', Validators.required) }),
    step5: new FormGroup({ confirm: new FormControl(false, Validators.requiredTrue) })

Interview Q&A — RxJS vs Signals

RxJS vs SignalsHard
With Signals in Angular 17+, is RxJS obsolete? Give a comprehensive answer covering where each belongs.
▼

Short answer: No, RxJS is not obsolete

Signals and RxJS solve fundamentally different problems. The Angular team has explicitly stated they do not plan to remove RxJS. The two are designed to work together via toSignal() and toObservable().

Keep using RxJS for:

  • HTTP with cancellation: switchMap auto-cancels the previous in-flight request when a new value arrives. Signals have no equivalent for this pattern.
  • Debounce / throttle: debounceTime(300), throttleTime(1000) for search-as-you-type, scroll handlers, button anti-double-click.
  • Complex stream coordination: combineLatest([a$, b$]), forkJoin([req1$, req2$]), withLatestFrom(store$).
  • Retry logic: retry({ count: 3, delay: 1000 }), exponential backoff with retryWhen.
  • WebSocket streams: long-lived multi-value streams over time.
  • Error recovery pipelines: catchError(() => of(fallback)), onErrorResumeNext.

Replace with Signals:

  • Local component state (toggles, tabs, loading flags, form values) — previously BehaviorSubject
  • Derived values — previously combineLatest + map + distinctUntilChanged, now computed()
  • @Input() bindings — now signal inputs
  • Store patterns without async coordination — plain signal services
  • Template binding — previously | async pipe, now signals read directly

The canonical bridge pattern

export class UserDashboardComponent {
  // 1. Signal for local state
  userId  = signal<string | null>(null);
  loading = signal(false);

  // 2. RxJS pipeline for async coordination + cancellation
  private user$ = toObservable(this.userId).pipe(       // signal → RxJS
    filter(id => !!id),
    tap(() => this.loading.set(true)),
    switchMap(id => this.userService.get$(id!)),        // cancels prev request
    tap(() => this.loading.set(false)),
    catchError(() => { this.loading.set(false); return of(null); })
  );

  // 3. Signal for template — no async pipe needed
  user = toSignal(this.user$, { initialValue: null });  // RxJS → Signal

  // 4. computed() for derived UI state
  displayName = computed(() => this.user()?.name?.toUpperCase() ?? 'Guest');
}

This pattern gives you the best of both worlds: ergonomic signals in templates, full RxJS power for async coordination.
